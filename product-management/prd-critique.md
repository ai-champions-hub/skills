# PRD Critique

> Existing PRD → structured feedback with specific issues, rubric scores, and next steps.

## Context

You have a PRD (yours or someone else's) and want rigorous, structured feedback before it goes to engineering. This skill reviews the document against a quality rubric — clarity of user problem, measurability of success metrics, scope definition, edge cases, and more. Critique-first instead of generate-first; rare and high-value.

Based on the Digidai Product Manager Skills framework, which embeds 40+ PM frameworks and an opinionated interaction style that labels assumptions and names tradeoffs.

## Instructions

1. I will paste a PRD or point you to a document.
2. Review it against these quality dimensions:

**Rubric (score each 1-5):**

| Dimension | What to evaluate |
|-----------|-----------------|
| Problem clarity | Is the user problem specific, evidence-backed, and free of solution smuggling? |
| User definition | Are users described as specific roles with context and motivation, not generic "users"? |
| Goals & metrics | Are goals measurable with number, direction, and timeframe? Are leading and lagging indicators separated? |
| Scope control | Are non-goals explicit? Is there clear in/out scope? Could any requirement be cut without losing the core value? |
| Requirements quality | Are requirements prioritized (P0/P1/P2)? Do P0s map directly to the problem statement? |
| Edge cases | Are failure modes, empty states, and error scenarios addressed? |
| Acceptance criteria | Are criteria testable? Given/When/Then format or equivalent? |
| Tradeoffs | Are tradeoffs named explicitly, not hidden behind "we'll figure it out"? |

3. For each dimension:
   - Give a score (1-5)
   - Cite the specific text that's problematic (quote it)
   - Explain what's wrong and why it matters
   - Suggest a concrete fix

4. Produce a summary:
   - **Overall assessment**: READY / NEEDS WORK / MAJOR REWORK
   - **Top 3 issues** (in priority order)
   - **Strengths** (what's working well — be specific)
   - **Assumptions to validate** (tag each with `[assumption]`)
   - **Recommended next step** (one action, not a list)

5. **Anti-patterns to flag:**
   - Metrics Theater — metrics that sound good but don't drive decisions
   - Feature Factory — requirements list without problem statement
   - Solution Smuggling — the "problem" is actually a proposed solution
   - Confirmation Bias — cherry-picked evidence supporting a predetermined conclusion
   - Scope Creep — P2 requirements that should be a separate PRD

6. End with a completion status: `DONE`, `DONE_WITH_CONCERNS`, `BLOCKED`, or `NEEDS_CONTEXT`.

## Example

### Input

*(Any PRD document — internal, public, or generated by the write-spec skill.)*

### Output

**Overall: NEEDS WORK (3.2/5)**

| Dimension | Score | Key issue |
|-----------|-------|-----------|
| Problem clarity | 4 | Clear problem, but "users struggle" needs data |
| User definition | 2 | "Users" appears 12 times with no role specificity |
| Goals & metrics | 3 | Goals exist but no baselines or timeframes |
| Scope control | 2 | No non-goals section; 3 requirements could be separate features |
| Requirements quality | 4 | Good MoSCoW split, P0s are solid |
| Edge cases | 1 | No error states, no empty states, no offline scenario |
| Acceptance criteria | 3 | Present but not testable — "works well" isn't a criterion |
| Tradeoffs | 2 | Two architecture decisions hidden in assumptions |

**Top 3 issues:**
1. No non-goals → scope will creep at sprint planning. Add 3-5 explicit exclusions.
2. "Users" undefined → engineering will build for the wrong persona. Replace with named roles.
3. Zero edge cases → QA will find them in production. Add failure modes for each P0.

**Recommended next step:** Add non-goals section and replace every instance of "users" with a specific role before the next review.

Status: `DONE_WITH_CONCERNS`
